## 0. 全体像とMVPの線引き

### 0.1 コアループ（MVPで必ず動く範囲）
開演（幕間演出）→ 参加者を組に自動分割 → 【周回ループ】各組が順番に舞台へ（お題発表 → 90秒の持ち時間で各演者が最大5回答、送信ごとに「タイマー停止→お題再提示→回答表示→10秒採点→再開」の審査サイクル → 次の組）→ 設定周回数ぶん繰り返し → 最終結果（個人1〜3位の表彰）→ アーカイブ生成

### 0.2 MVPに含める / 含めない
| 区分 | 含める（MVP） | 含めない（将来拡張） |
|---|---|---|
| 開催 | 運営が手動でライブを立てる／定時開催 | 完全自動スケジューラ、複数ライブ同時並行 |
| 認証 | ログイン（X/メール）を基本導線。ゲスト参加の可否は保留（§7） | フルSNS機能、DM |
| 役割 | 組分けによる回答者/観客の自動ローテ | 立候補制、指名制 |
| お題 | 運営が事前登録した固定お題セット（text / image_caption） | ユーザー投稿お題、AI生成 |
| 審査 | 1審査員→1回答につき0/1/2点、10秒採点、笑いエフェクト | コメント審査、投げ銭 |
| 進行 | 送信ごとの審査サイクル＋排他制御、司会コントロールパネル（半自動）、周回制 | 完全自動進行のみ運用 |
| 経済 | 消費用ポイント＋表彰ボーナス、熟練度メーター→段位の2軸 | ガチャ本実装（抽象設計のみ）、シーズン制、リーグ戦 |
| コレクション | アイテム種類/レアリティ/排出テーブルの抽象データモデル | 具体的なキャラ/衣装/背景の中身、ガチャ演出 |
| 演出 | 幕間演出（暗転→タイトル）、笑いエフェクト、拍手/テンプレツッコミ | 高度な舞台演出・3D・BGM本制作 |
| アーカイブ | リザルト閲覧＋Xシェア（OGP） | 動画化、O-OGIRI連携シェア |
| モデレーション | NGワード辞書＋司会の事後対処 | AI文脈判定、自動BAN |

### 0.3 差別化ポイント（競合比較）
- **大喜利オンライン**：常時マッチングの対戦バトル（座布団制、勝敗が主）。
- **大喜利茶屋**：随時プレイの4分リアルタイム大喜利。
- **本アプリ**：**「定時に一斉開催されるライブイベント」という祭り性**が核。X告知連動の開演、客席⇄舞台のロール入れ替え、送信ごとの寄席的な「間」と採点演出、笑いエフェクト、その日限りの表彰を体験の中心に据える。姉妹企画とも棲み分け（SNS＝ストック／育成＝ソロ育成／道場＝ライブ瞬発）。

## 1. 進行フォーマット詳細（全面改訂）

### 1.1 1組の進行モデル（コア）
1組は目安**5人**。各組の**持ち時間は90秒**。この90秒は「回答を考えて送信する側」の時間であり、審査の時間は含まない（別枠加算）。

- 各回答者は持ち時間内に**最大5回答**まで投稿できる。
- 回答者が「送信」を押すたびに、以下の**審査サイクル**が1件ずつ発生する：
  1. 全体タイマーが**一時停止**する。
  2. **お題が再提示**される（音声 or 無音の演出）。※お題自体は変わらない。同じお題の演出的な再掲示。
  3. **一呼吸置く**（約0.7秒、画面には何も表示しない静止区間）。前の回答が表示されていた場合は、いったん完全に消えてからこの間に入る。
  4. その回答者が今送信した回答が表示される。
  5. 客席（審査員）が**10秒以内**に採点する（採点方式は §4）。**採点権を持つ全員が投票し終えたら、10秒を待たずに即座に締め切る（早期確定）**。
  6. 採点が終わると全体タイマーが**再開**する。
- キューに複数の送信が溜まっている場合も、1件の審査サイクルが終わるたびに必ず③の「一呼吸」を挟んでから次の回答を表示する（回答同士が間断なく連続表示されて忙しなく見えることを防ぐ、2026-07-14追記）。
- このサイクルは**誰かが送信するたびに毎回**発生する（1組あたり最大 5人 × 5回答 ＝ **最大25回**）。
- **排他制御**：サイクル中（誰かの回答が表示〜採点完了〜一呼吸まで）は、**送信そのものができない**（§1.4）。テキストの入力・編集自体はサイクル中も常に可能で、サイクルが終わって送信可能になった時点で送信ボタンが再び押せるようになる（2026-07-14改訂：送信して審査待ちのまま予約しておく仕組みは、他人の回答と被っても取り返しがつかないため廃止）。
- **早期終了（確定）**：組の全員が最大回答数（5回）を使い切った時点で、持ち時間の残りを待たずに `answering` を終了し `group_result` へ進む。持ち時間はあくまで「回答を出し切るための上限」であり、余らせて待つ必要はない。

### 1.2 1組あたりの所要時間
```
1組の実所要 ＝ 持ち時間90秒 ＋ Σ(発生した審査サイクルの時間)
　審査サイクル1回 ≒ お題再提示演出(可変) ＋ 採点10秒 ＋ 前後の遷移
```
- 持ち時間の90秒は「送信していない待機・思考の時間」で消化され、送信するとその間はタイマーが止まる。
- したがって最大25回の審査サイクルが起きれば、1組の実時間は90秒＋（25×十数秒）まで伸びうる。尺の見積り・司会の進行判断に用いる。

### 1.3 周回制（roundsPerLive）
- 全組が1回ずつ舞台に立つのを**1周**とする。
- 何周するかは運営が調整できる（設定値 `roundsPerLive`）。**デフォルト目安は2周**だが固定しない。
- 周回ぶん、組ごとに**別のお題**が必要（例：2周なら各組2問ずつ用意）。→ お題セットは「組数 × 周回数（＋予備）」ぶん仕込む。
- 舞台に立つ順序：`round` ごとに全組が `group.order` 順で登壇する（例：2周・3組なら A→B→C→A→B→C）。

### 1.4 送信の排他制御（実装要件）
- サーバは各組の登壇中、**単一の「審査中スロット」**を持つ。スロットが空のとき（誰の回答も表示・採点されていない、一呼吸中でもない状態）のみ送信を受理する。
- 複数人がほぼ同時に送信した場合、**サーバ受信順**で1件ずつ処理する。ただし2026-07-14改訂により、送信できるのはスロットが空の瞬間のみとし、「送信して次の順番を予約しておく」ことはできない。送信ボタンはスロットが空でない間は非活性表示になり、押せない（例：「審査中…少し待ってね」）。
- 非活性化の対象は**送信ボタンのみ**。テキスト入力欄は採点フェーズ中も常に操作可能で、回答回数の上限に達した回答者のみ入力欄を閉じる。これにより、他人の回答が表示されている間に内容を確認してから、被らない回答を選んで送信できる。
- 持ち時間タイマーは**サーバ基準**で管理し、審査サイクル中は**サーバ側でpause/resume**する（クライアント時計は信用しない）。持ち時間の残り表示は、フェーズや採点中かどうかに関わらず常に見える状態を保つ（採点中は一時停止していることが分かる表示に留め、非表示にはしない）。

### 1.5 フェーズ一覧（デフォルト値）
| フェーズ | 状態キー | 時間 | 内容 |
|---|---|---|---|
| 受付 | `lobby` | 定刻〜+5分 | 参加者集合。人数カウント表示。 |
| 幕間 | `interlude` | 10〜20秒 | 暗転 → タイトル表示（§12）。開演演出。 |
| 開幕 | `opening` | 30〜60秒 | ルール説明・組分け結果通知。 |
| お題発表 | `topic_reveal` | 15〜30秒 | その組・その周のお題を全員に提示。舞台メンバー紹介。 |
| 回答（持ち時間） | `answering` | **90秒**（審査中はpause） | 舞台の組が最大5回答ずつ投稿。送信ごとに審査サイクル。 |
| 審査サイクル | `judging`（answering内のサブ状態） | 送信ごと・**10秒採点** | お題再提示 → 回答表示 → 採点 → 再開。 |
| 組結果 | `group_result` | 20〜40秒 | その組のこの周のスコア上位を発表。 |
| （組ループ × 周回ループ） | | | `roundsPerLive` ぶん繰り返し |
| 最終結果 | `final_result` | 2〜3分 | 個人1〜3位の表彰、ベストアンサー、成績カード。 |
| 閉幕 | `closed` | — | アーカイブ生成、Xシェア導線。 |

### 1.6 スコア表示タイミング（推奨案）
- **採点結果（0/1/2点の集計）は、その回答の採点完了時（10秒締切後）に確定表示**する。ライブ感のため合計スコアはその場で見せる。
- **過半数が2点**なら採点完了と同時に**笑いエフェクト**を舞台に発生させる（§4.3）。
- 回答カードの並びはスコア降順に並べ替えず、**投稿順**とする（煽りは組結果まで伏せる）。運用で荒れたら切替可能な設計（`revealScoreDuringLive` フラグ）。

### 1.7 ライブの締め（推奨案）
全周終了後、最終結果で以下を段階発表：
1. **本日のベストアンサー**：全回答中、単一回答で最高スコアの一撃（回答文をどーんと表示）。
2. **個人表彰 1〜3位**：個人合計スコア上位3名を、3位→2位→1位の順で発表（§5.4のボーナス付与）。
3. **総合ランキング**：個人合計スコアの一覧（全員ぶんスクロール可）。
- 1〜3位・ベストアンサーは通算実績（熟練度メーター・カウント）にも加算（§5・§6）。
- 全員に「あなたの成績カード」（獲得スコア・順位・段位変化・獲得ポイント）を提示 → Xシェアの起点にする。

## 2. 人数変動時のグループ分け

### 2.1 標準編成と人数連動
1組は目安**5人**（許容 3〜6人）、組数は**2〜4組**を基本に、集まった人数 N から自動編成する。

| 参加人数 N | 編成 | 備考 |
|---|---|---|
| N < 6 | **不成立**（§10のフォールバックへ） | 客席が薄すぎ審査が成立しないため。 |
| 6〜9 | 2組（3〜5人×2） | ミニ開催。 |
| 10〜18 | 3組（目安5人前後） | 標準フォーマット。 |
| 19〜24 | 4組 | 舞台数を増やして尺を維持。 |
| N ≥ 25 | 4組（各上限6人＝最大24人）＋**観客専任枠** | あぶれた人は観客専任（§3.3）。 |

- **最低開催人数：6人**（推奨）。ある組が舞台のとき客席が最低3人確保でき、採点が成立する最小構成。
- **均等割り**：N を組数で割り、余りは前の組から1人ずつ配分（例：13人・3組 → 5/4/4）。
- **周回制との関係**：組編成は開幕時に1回だけ確定し、全周を通して固定（周ごとに組は変えない）。

### 2.2 端数・欠員への耐性
- 開幕後に離脱者がいても組は再編成せず**そのまま進行**（ライブなので止めない）。舞台の組が1人になっても回答は成立させる。
- 舞台の組が全員離脱した稀ケースは、司会がその組をスキップ（`group.status = "skipped"`）できる。
- 周の途中で登壇中の組が薄くなった場合も、司会判断でそのターンをスキップ可能。

## 3. 役割決定方法

### 3.1 基本方針（推奨案）
**「参加＝どこかの組に入る＝いつか必ず舞台に立つ」を基本とし、組分けは完全ランダム自動割当。**
- 全員が回答者と観客の両方を1ライブで体験するのが企画の核。ランダムなら毎回違う顔ぶれと当たり公平。
- 立候補制・先着制は組の偏り・早押し疲れを生むため不採用。

### 3.2 舞台の順番
- 組の**出番順もランダム**に決定（`group.order`）。周回ごとに同じ順序で登壇する。
- 「先攻/後攻」の有利不利はお題が組ごと・周ごとに違う（§8）ため実質フラット。
- 各参加者には開幕時に「あなたは◯組・出番◯番目」を通知（画面L2）。

### 3.3 回答者が多すぎ／少なすぎる場合
- **多すぎ（N≥25）**：組人数上限（6人）超過ぶんは**観客専任枠**へ。観客専任者には「次回ライブの舞台優先枠」を付与（`priorityNextLive = true`）。
- **少なすぎ**：本アプリは「参加＝舞台に立つ」が原則。純観戦したい人向けに**見学モード**（§7.3）を別途用意。見学者は組にカウントしない。
- 受付時に**「舞台参加／見学のみ」を選べる**。舞台参加者だけで §2.1 の組編成を行う。

## 4. 審査方式：スコアリング（全面改訂：いいねトグル → 0/1/2点）

### 4.1 採点の基本ルール
- 客席の審査員は、審査サイクルで表示された各回答に対し、**0点・1点・2点**の3段階でスコアを付ける（**10秒以内**、§1.1）。
- 採点対象は「今まさに審査サイクルに乗っている1回答」のみ。1審査員は1回答につき1採点（0/1/2のいずれか）。
- 未採点のまま10秒が経過した審査員の票は **0点扱い**（＝未投票＝加点なし）とする。
- **早期確定（確定）**：採点権を持つ審査員全員が投票し終えた時点で、10秒を待たずに即座にそのサイクルを締め切り、次（タイマー再開・次の送信受付）へ進む。
- 自分自身が舞台に立っている間は採点しない（舞台側は審査UIを持たない）。
- 見学者の採点可否は運用フラグ（§7.3、初期は不可推奨）。

### 4.2 スコアの集計
- ある回答の獲得スコア `answerScore` ＝ その回答に付いた全審査員の点数の合計（0/1/2の総和）。
- 演者のライブ内スコア ＝ 自分の全回答の `answerScore` 合計。
- 通算スコアは全ライブ累積で加算（§6 の熟練度メーターへ反映）。
- **旧「いいね合計ベース」の算出式は廃止**し、上記「獲得した0/1/2点の合計」ベースに一本化する。

### 4.3 笑いエフェクト（過半数2点で発生）
- ある回答の採点締切時、**その組が舞台のときの客席（審査員）人数の過半数**が2点をつけていたら、舞台に**「笑いエフェクト」**を発生させる。
  - 判定：`(2点をつけた審査員数) > floor(該当時点の審査員数 / 2)`。
  - 「審査員数」はその組登壇中に採点権を持つ人数（見学者を含めるかは §7.3 のフラグに従う）。
- 笑いエフェクトは演出のみ。スコアそのもの（§4.2）とは独立で、二重加点はしない。

### 4.4 演出専用アクション（スコア非加算）
- 審査員は**「回答中（審査サイクル）以外」のタイミング**で、以下を自由に押せる。**ポイント・スコアには一切影響しない純演出**：
  - **拍手エフェクト**（連打可）。
  - **定型ツッコミワード（テンプレボタン）**：新規追加。「なんでやねん」「そうはならんやろ」等の短文スタンプを画面に飛ばす。
- これらは審査サイクル中（採点中）は誤操作・妨害防止のため非活性にしてよい。

### 4.5 UI要件（改訂：画面固定・使い捨て表示、2026-07-13確定）
- **画面固定**：ライブ画面はブラウザのビューポート内に常に完全に収まる（スクロール不要）。縦スクロール・横スクロールが発生するレイアウトは不可。
- **使い捨て表示（確定）**：過去の回答を一覧・タイムライン形式で溜め込む表示は**すべて廃止**する。「流れた回答一覧」のようなリストは持たない。
  - 回答表示のライフサイクル：①回答が投稿されたら画面中央に大きく表示する → ②審査が終了（採点確定 or 一定時間経過）したら、その回答は即座に画面から消去する。常に「今まさに審査されている1件」だけが画面に存在する状態を保つ。
- 客席（審査員）画面：審査サイクル中は「再提示されたお題＋対象回答（中央に大きく）＋0/1/2の採点ボタン＋残り10秒」を表示。サイクル外は回答の一覧を出さず、次の登壇を待つ簡潔な待機表示＋拍手/テンプレツッコミのみ。
- 舞台（回答者）画面：残り持ち時間、回答入力欄（最大5回・残り回数表示、審査サイクル中も常に入力可）、送信ボタンの非活性表示（「他の回答を審査中」、審査サイクルが終わるまで送信不可）、自分の回答が得たスコア。他メンバーの回答も同様に「今審査中の1件」を中央に大きく表示する形で見える（一覧化しない）。
- リアルタイム反映：採点締切と同時に全クライアントへ確定スコア・笑いエフェクトを配信。確定表示後、短い演出を経て画面から消去する。

### 4.6 不正防止
- `Score` は `(answerId, judgeParticipantId)` でユニーク制約 → 二重採点不可（同一審査員は上書きのみ）。
- 大量アカウント生成対策は §9 のレート制限・1セッション1参加で担保。
- 採点は10秒枠内のみ受理。枠外の採点は破棄（サーバ時刻判定）。

## 5. 経済：ポイント・ガチャ・コレクション（新規／全面改訂）

### 5.1 2軸構成（最重要）
本アプリの数値経済は**2軸**で構成する。混同しないこと。

| 軸 | 名称 | 性質 | 用途 | 増える条件 |
|---|---|---|---|---|
| ① | **ポイント（消費用通貨）** | 使うと減る残高 | キャラ入手／背景の柄変更／ガチャ | 表彰ボーナス、参加ボーナス等 |
| ② | **熟練度メーター（経験値）** | 累積して減らない | 段位（腕前の称号）を上げる | 表彰実績、獲得スコアの蓄積 |

- ①は「使って楽しむ」ためのもの、②は「腕前を示す」ためのもの。片方を消費しても段位は下がらない。

### 5.2 ポイント（消費用通貨）
- **獲得**：ライブ表彰（1〜3位、順位別＝§5.4）、参加ボーナス（皆勤で少額）、（将来）デイリーミッション等。
- **消費用途**：
  - キャラクター（アバター的存在）の入手・変更。
  - プロフィール背景の柄変更などのカスタマイズ。
  - ガチャ（§5.3）を引く。
- ポイント残高は `User.points` として永続管理。ライブ中の獲得は `closed` 時に確定加算。

### 5.3 ガチャ・コレクション（抽象設計＋具体例）
- ポイントを消費してガチャを引き、**キャラの衣装・アイコンパーツ等をランダム入手**（モバゲー／ソシャゲ的コレクション性）。
- 本仕様では**システム側の抽象構造**（§5.3.1）と、**具体的なラインナップ・排出率・段位連動方針の案**（§5.3.2、2026-07-12追記）の両方を定義する。ラインナップの最終的なビジュアル制作はdesign-deptで別途進める前提。
- 所有・装備は `UserInventory` で管理（所有アイテムID配列＋装備中の割当）。
- 演出（ガチャ画面・排出アニメ）はMVP対象外。まずはデータ構造とポイント消費フローのみ。

#### 5.3.1 抽象構造
- 抽象化する要素：
  - **アイテムの種類**（`type`：costume / icon_part / bg_pattern / character …拡張可能）。
  - **レアリティ**（`rarity`：N / R / SR / SSR …）。
  - **排出テーブル**（`GachaTable`：どのアイテムがどの重みで出るか）。
- データモデルは §15 の `CollectionItem` / `GachaTable` / `UserInventory` を参照。

**④ レアリティ別排出率案**

一般的なソシャゲの相場（4段階）を踏まえた案：

| 案 | N | R | SR | SSR | 傾向 |
|---|---|---|---|---|---|
| A：カジュアル寄り | 50% | 35% | 12% | 3% | 引きやすさ優先、コレクション欲は弱まりやすい |
| B：バランス型（推奨） | 55% | 33% | 10% | 2% | 標準的なソシャゲ相場、渋さと引きやすさの中間 |
| C：渋め・レア感重視 | 60% | 30% | 8% | 1% | SSRの特別感は強いが、カジュアル層の継続意欲が下がりやすい |

- **推奨：B案（N55% / R33% / SR10% / SSR2%）**。
  - 理由：ガチャ1回100pt・1位表彰300pt（ガチャ3回分）という消費設計（§5.4・§5.5）から、参加を重ねれば無理なく複数回引ける前提。渋すぎる排出率（C案）だとカジュアル参加層の「引いても外れが多い」体験が続き、企画の「置いていかれない」哲学（§6.1・企画書）とややズレる。かといってA案では長期継続時にSSRの価値が薄れやすい。B案が両立点。
  - `GachaTable.entries[].weight` は type（costume / icon_part / bg_pattern）ごとに個別テーブルを作るか、全typeまとめた1テーブルで管理するかは実装判断。まずは**type横断で1本の `GachaTable`**とし、同一レアリティ内は等確率抽選とする設計を初期案とする。
  - 天井（保証）システム（例：N回以上SSR未排出でSSR確定）はMVP対象外の将来検討事項とする。

**⑤ 段位・ライブ実績との連動有無**

- **推奨：連動なし（全ユーザー同一の排出率で確定）**。
- 理由：
  - §6.4で「メーターの実数・通算スコアの実数は控えめに、段位という“称号”で見せる」「競争の刺々しさを抑える」方針を明記済み。ガチャ排出率まで段位に連動させると、上位段位ほどコレクションでも有利になり、初心者との体験差がスコア以外の軸にも波及して方針と矛盾する。
  - 企画書の「負けても通えば熟練度メーターは上がる」「置いていかれない」哲学とも、ガチャは全員フラットな方が整合する。
  - 実装・運用コストも「排出率が1本のテーブルで済む」ためシンプル。
- **デメリットへの対策（代替インセンティブ案）**：段位上昇のコレクション面での実利が薄くなる点は、ガチャの排出率をいじる代わりに以下で補う：
  - 段位到達時に**記念ガチャ無料チケットを1枚付与**（既存のガチャ抽選テーブルをそのまま利用、確率はいじらない）。
  - 段位に応じた**専用アイコン枠・称号背景**（ガチャ対象外・自動付与のレアリティ表現）を別途用意し、「引いて当てるコレクション」と「積み上げて得る称号演出」を区別して両立させる。
  - この代替案は将来拡張として`§17`（実装優先順位）の「（将来）ガチャ/コレクション本実装」フェーズで扱う。

### 5.4 表彰ボーナス（順位別、初期値確定）
- 最終結果で個人1〜3位を表彰。**順位に応じてボーナスポイントの量が変わる（1位が最も多い）**。
  - **初期値**：1位 = 300pt、2位 = 200pt、3位 = 100pt、参加賞（4位以下 全員）= 30pt、ベストアンサー = +100pt（順位ボーナスとは別枠で加算）。`bonusByRank` として設定値化し、運用データを見てチューニングする。
- 同時に**熟練度メーター**にも順位別・スコア別の経験値を加算（§6.2）。
- ベストアンサー獲得者にも追加ボーナス（ハレの実績）。

### 5.5 消費目安（参考値）
ポイントの「使いでのある額」を作るための、ガチャ・カスタマイズの消費目安（初期値、要チューニング）：
- ガチャ1回：100pt
- コスチューム個別購入：150〜300pt
- 背景柄変更：100pt

1位を獲得すればガチャ3回分、参加を重ねるだけでもいずれガチャを引ける、という消費リズムを想定。

## 6. 段位：熟練度メーター（改訂：8段階の寄席モチーフ呼称に確定）

### 6.1 メーターと段位の関係
- **熟練度メーター（`masteryMeter`）**：表彰実績や獲得スコアの累積で貯まる経験値。**減らない**。
- メーターが一定量貯まると**段位**（腕前の呼称）が上がる。段位はポイント残高とは無関係。

### 6.2 メーターの加算（初期値確定）
- **参加ボーナス**：ライブ1回につき +10（皆勤でも確実に貯まる＝“置いていかれない”哲学。上位段位を目指さない層でも参加を続ける限り着実に前進する）。
- **獲得スコア比例**：そのライブでの個人合計スコア（`Participant.scoreReceived`）を**そのまま1:1でメーターに加算**（複雑な係数を挟まず直感的にする）。
- **順位ボーナス**：1位 +100 / 2位 +60 / 3位 +30（参加賞のみの回は加算なし、参加ボーナス+10は別途つく）。
- **ベストアンサー**：+50（順位ボーナスとは別枠で加算）。
- 上記はすべて `masteryGain` の初期値として設定値化し、運用データを見てチューニングする前提。

### 6.3 段位の呼称（確定：8段階）
実在する落語家・芸人の位（前座・二ツ目・真打）をモチーフに、「見習い」始まり・「達人」終わりで挟んだ8段階で確定（2026-07-12）。「ネオ寄席」の世界観と直結する呼称。

| 段 | 呼称 | イメージ |
|---|---|---|
| 1 | 見習い | まだ右も左もわからない新人 |
| 2 | 前座 | 寄席で最初に正式な出番をもらえる位 |
| 3 | 二ツ目 | 一人前として認められ始める中堅の入口 |
| 4 | 真打 | 落語界で最高位とされる、独り立ちした演者 |
| 5 | 花形真打 | 真打の中でも人気・実力で頭角を現す存在 |
| 6 | 大看板 | 寄席の顔として名前が看板に大きく載る大物 |
| 7 | 名人 | 業界内外から認められる名手 |
| 8 | 達人 | 誰もが認める頂点、道場の伝説的存在 |

- `rank` の値は内部的には `"minarai" | "zenza" | "futatsume" | "shinuchi" | "hanagata_shinuchi" | "ookanban" | "meijin" | "tatsujin"` のようなキーで管理し、表示名は上記の日本語呼称を用いる。

### 6.3.5 昇段閾値（初期値確定）
`masteryMeter`の累積値が以下のラインを超えると昇段する（`rankThresholds`として設定値化、運用データを見てチューニングする前提）。

| 段 | 呼称 | 累積閾値 |
|---|---|---|
| 1 | 見習い | 0（スタート） |
| 2 | 前座 | 150 |
| 3 | 二ツ目 | 400 |
| 4 | 真打 | 900 |
| 5 | 花形真打 | 1,800 |
| 6 | 大看板 | 3,200 |
| 7 | 名人 | 5,500 |
| 8 | 達人 | 9,000 |

想定ペース：参加ボーナス+獲得スコアで1ライブあたり平均40前後のメーター獲得を仮定すると、「前座」到達は数回の参加で、「達人」到達は毎日〜週数回のペースで長期間通い続けた末にようやく届く難易度感を狙う（“真に極めた者だけが辿り着く頂点”という達人の位置づけを保つ）。序盤の段位間隔を狭く・終盤を広くする指数的な刻みにしているのはこのため。

### 6.4 表示ルール
- 段位はマイページ（L8）とライブ中の名札に表示。
- **メーターの実数・通算スコアの実数は控えめに、段位という“称号”で見せる**（競争の刺々しさを抑える）。

## 7. アカウント・認証（改訂：ログイン必須・ゲスト参加なしに確定）

### 7.1 方針：ログイン必須（確定）
**ゲスト参加は一旦なし。ログイン（アカウント登録）必須とする**（2026-07-12確定）。

- 理由：経済システム（ポイント/ガチャ/段位）が本アプリの魅力の中核であり、全参加者にポイント・熟練度メーターが確実に貯まる状態を優先。告知がXのため、X連携を第一導線に置く。
- `guestJoinAllowed` はコード上は残すが**既定値・現状運用ともに `false` 固定**。将来、集客戦略上ゲスト参加を解禁したくなった場合の切替スイッチとしてのみ設計に残す（現時点で有効化する予定はない）。

### 7.2 独立方針
- 姉妹企画（O-OGIRI／育成）とは**アカウントを共有しない完全独立**。将来連携の余地は残す。

### 7.3 見学モード（確定：見学者は見るだけ）
- 「見学のみ」でライブを覗ける（§3.3）。**見学者は採点に参加できない（確定、2026-07-12）**。採点は組に所属する客席メンバーのみが行う。`spectatorCanScore` は既定値・現状運用ともに `false` 固定。
- 笑いエフェクトの過半数判定（§4.3）の母数にも見学者は含めない。

## 8. お題の投入方法（改訂：フォーマット拡張を追加）

### 8.1 方針：事前登録＋司会の半自動出題
- 運営（司会）はライブごとに**「お題セット」**（**組数 × 周回数** ＋ 予備1〜2問）を事前登録。
- ライブ中、**組が舞台に上がるたびに、その組・その周のお題が自動で紐づく**（`group.order` × `round` 順に割当）。
- フェーズ送りは**司会が「進行」ボタンで手動送り**できる半自動方式。持ち時間タイマー・採点タイマーは自動だが、トラブル時に司会が介入できる保険を持たせる。
- 完全自動進行モード（`autoAdvance = true`）も選べるが、MVP運用は**司会手動を推奨**。

### 8.2 お題は組ごと・周ごとに別（確定事項）
- 各組・各周で別お題（同一お題だと後攻がパクリ・改変で有利になるため）。
- お題プールは再利用可（別日のライブで使い回し可）。使用履歴 `usedInLiveIds` で被り回避を支援。

### 8.3 お題フォーマット（MVPはtextのみで確定）
- **MVPでは `text`（文章のみ）のみを実装**（2026-07-12確定）。`image_caption` 等は当面作らない。
- ただし `Topic.format: "text" | "image_caption" | ...` のフィールド自体は**将来拡張を見込んで最初から持たせておく**（型・データモデルは拡張可能な設計のまま、実装範囲だけMVPをtextに絞る）。
  - `text`：本文テキストのみ（MVP実装対象）。
  - `image_caption`：画像URL＋一言（キャプション）。**将来拡張、MVP対象外**。
  - 将来例：`video`、`audio`、`fill_in_blank`（穴埋め）等を追加可能。
- 再提示演出（§1.1-2）は現状 `text` のみを前提に設計してよい（画像対応時に演出を拡張）。

## 9. モデレーション（最低ライン）

- **方針**：軽い下ネタOK・露骨な卑猥/差別/倫理NG不可。ライブなので**事前検閲はしない**。
- **自動フィルタ（投稿時）**：NGワード辞書をJSON管理。`hardNG`＝投稿ブロック（本人に警告）／`softNG`＝通すが `flagged=true` で司会画面に強調。伏字・記号回避は正規化してから照合。
- **司会（運営）権限**：
  - 特定回答の**即・非表示**（客席から消える／スコア無効化）。
  - 参加者の**強制退室（キック）**＋同一ライブ再入場ブロック。
  - 荒れた場合の**ログイン必須モード切替**（§7.1 `guestJoinAllowed=false`）。
- **採点不正対策**：`Score` の `(answerId, judgeParticipantId)` ユニーク制約で二重採点を無効化。加えてゲスト量産対策として**1セッション1参加・IP/デバイス単位の受付レート制限**（ゲスト許可時）。
- **通報**：客席から回答に通報ボタン（理由選択）。閾値（例3件）で司会レビューキューに強調。ライブ中は司会判断、事後はアーカイブから非表示化。

## 10. 人が集まらない場合

- 定刻から**受付5分**（`lobby`）。
- 受付終了時点の人数で判定：
  | 集まった人数 | 挙動 |
  |---|---|
  | N ≥ 6 | §2.1 のフォールバック編成で**そのまま開演**（少人数でも決行）。周回数は司会が減らす判断も可。 |
  | N < 6 | **不成立**。「本日は人数不足のため中止／次回◯時に順延」を自動告知（画面＋Xシェア文面生成）。集まった数人には**エキシビション（1組だけの練習会）**を選択肢提示可（`exhibitionMode`）。 |
- 司会は受付を**手動延長**（例：+3分）できる。告知時間より多少ズレても開演を優先。

## 12. 演出・世界観（新規）

### 12.1 方向性
- 舞台美術・UIは**「芸人がネタをする舞台／寄席」**をイメージ。演出・世界観に**強くこだわる**要件。
- 具体的なビジュアル仕様（色・書体・舞台セット・キャラ造形）は**design-deptとの別セッションで詰める**前提。本書では要件と状態・画面の存在を定義する。

### 12.2 幕間演出
- ライブ開始前に**「幕間」演出**を挟む：**暗転 → タイトル表示**（→ 開幕へ）。
- 状態キー `interlude`（§1.5・§13）として進行に組み込む。周と周の切れ目、組の切り替え時にも短い幕間（暗転＋次組紹介）を挟めるよう拡張可能に。

### 12.3 ライブ中の演出要素
- **お題再提示演出**（送信ごと、§1.1）：音声 or 無音でお題を再掲示し、審査の「間」を作る。
- **笑いエフェクト**（過半数2点、§4.3）：舞台に大きく発生。
- **拍手／テンプレツッコミ**（§4.4）：客席から飛ぶ純演出。
- **表彰演出**（§1.7）：3位→2位→1位の段階発表、ベストアンサーの大写し。

## 13. 進行フローの状態遷移（ライブ・ステートマシン）

```
[scheduled] ──定刻到達──▶ [lobby] ──受付締切&人数OK──▶ [assigning] ──▶ [interlude] ──▶ [opening]
   （告知済み）           （受付5分）   │            （組分け確定）  （暗転→タイトル） （組分け通知）
                                       └人数<6──▶ [aborted]（中止/順延）              │
                                                                                       ▼
                                            ┌──────────────── 周回ループ (round = 1..roundsPerLive) ───────────────┐
                                            │  ┌──────────── 組ループ (group.order = 1..組数) ────────────┐        │
                                            │  │  [topic_reveal] （その組・その周のお題）                 │        │
                                            │  │        ▼                                                 │        │
                                            │  │  [answering] 持ち時間90s（サーバでpause/resume）        │        │
                                            │  │     └▶ 送信ごとに [judging]：お題再提示→回答表示→10s採点  │        │
                                            │  │            → 笑いエフェクト判定 → タイマー再開            │        │
                                            │  │        ▼（持ち時間満了 or 全員5回答消化 or 司会送り）      │        │
                                            │  │  [group_result] （この周のこの組のスコア上位）           │        │
                                            │  └───────────────────────────────────────────────────────┘        │
                                            └────────────────────────────────────────────────────────────────────┘
                                                                    全周・全組終了 ▼
                                                              [final_result] ──▶ [closed]
                                                          (個人1〜3位表彰/ベスト)  (アーカイブ生成)
```

- **ライブの状態**：`scheduled | lobby | assigning | interlude | opening | in_progress | final_result | closed | aborted`
- **進行の現在位置**：`currentRound`（何周目）＋ `currentGroupOrder`（何組目）＋ `groupPhase: topic_reveal | answering | group_result`
- **answering中のサブ状態**：`judging`（審査サイクル中）と、対象 `answerId`・審査残り秒。
- 遷移トリガー：タイマー満了 or 司会の「進行」ボタン（§8、半自動）。
- タイマーは**サーバ基準時刻**で全クライアント同期。審査サイクル中は**サーバ側で持ち時間を停止/再開**。クライアント時計は信用しない。

## 15. データモデル概要（MVP）

エンティティ関係：**Live 1 —— * Group**、**Live 1 —— * Turn（組×周の1登壇） 1 —— 1 Topic**、**Live 1 —— * Participant**、**Turn 1 —— * Answer 1 —— * Score**、**User 1 —— * Participant**。

```ts
// ライブ（1回の開催イベント）
interface Live {
  id: string;
  title: string;
  scheduledAt: string;                 // 開演予定時刻
  status: "scheduled" | "lobby" | "assigning" | "interlude" | "opening"
        | "in_progress" | "final_result" | "closed" | "aborted";
  format: {
    answerSeconds: number;             // 各組の持ち時間。既定90
    maxAnswersPerPlayer: number;       // 1組内で1人が投稿できる最大数。既定3
    judgeSeconds: number;              // 審査サイクルの採点時間。既定10
    roundsPerLive: number;             // 周回数。既定2（固定しない）
    lobbySeconds: number;              // 受付。既定300（5分）
    minPlayers: number;                // 既定6
    maxPerGroup: number;               // 既定6（目安5）
  };
  topicSetId: string;                  // 使用お題セット
  hostId: string;                      // 司会（運営）
  currentRound: number | null;         // 現在の周（1始まり）
  currentGroupOrder: number | null;    // 現在舞台の組（1始まり）
  groupPhase: "topic_reveal" | "answering" | "group_result" | null;
  judging: {                           // answering中の審査サイクル状態
    answerId: string;                  // 採点対象の回答
    endsAt: string;                    // 採点締切（サーバ基準）
  } | null;
  answeringPausedRemainingMs: number | null; // 審査中に停止した持ち時間の残り
  phaseEndsAt: string | null;          // 現フェーズ終了時刻（サーバ基準）
  autoAdvance: boolean;                // 自動送りか司会手動か（既定false=手動）
  revealScoreDuringLive: boolean;      // ライブ中に確定スコアを見せるか（既定true）
  guestJoinAllowed: boolean;           // ゲスト参加可否（保留・運用切替）
  spectatorCanScore: boolean;          // 見学者の採点可否（既定false）
  createdAt: string;
}

// 参加者（あるライブへの1回の参加。ゲスト許可時はuserId無し）
interface Participant {
  id: string;
  liveId: string;
  userId: string | null;               // ログイン参加なら紐づく
  displayName: string;
  role: "player" | "spectator";        // 舞台参加 or 見学のみ
  groupId: string | null;              // 舞台参加者のみ所属（全周固定）
  scoreReceived: number;               // このライブでの獲得スコア合計（0/1/2の総和）
  priorityNextLive: boolean;           // 観客専任あぶれ者への次回優先
  isKicked: boolean;
  joinedAt: string;
}

// 組（開幕時に確定、全周固定）
interface Group {
  id: string;
  liveId: string;
  order: number;                       // 出番順（1..組数）
  memberIds: string[];                 // Participant.id の配列
  status: "waiting" | "on_stage" | "done" | "skipped";
}

// ターン（組 × 周 の1登壇。ここにお題が1つ紐づく）
interface Turn {
  id: string;
  liveId: string;
  groupId: string;
  round: number;                       // 何周目（1..roundsPerLive）
  order: number;                       // その周での登壇順（= group.order）
  topicId: string;                     // この登壇のお題（組ごと・周ごとに別）
  status: "waiting" | "topic_reveal" | "answering" | "group_result" | "done" | "skipped";
}

// お題（フォーマット拡張可能）
interface Topic {
  id: string;
  setId: string;                       // お題セットに属する
  format: "text" | "image_caption";    // 将来: "video" | "audio" | ... 拡張可
  body: string;                        // text本文 / image_captionの一言
  imageUrl?: string;                   // image_caption等で使用
  usedInLiveIds: string[];             // 被り回避の使用履歴
  tags: string[];
}

// 回答
interface Answer {
  id: string;
  liveId: string;
  turnId: string;                      // どの登壇（組×周）での回答か
  groupId: string;
  topicId: string;
  participantId: string;               // 回答者
  seq: number;                         // その人のその組での何回目（1..maxAnswersPerPlayer）
  body: string;
  scoreTotal: number;                  // 集計値（Scoreの点数合計 = 0/1/2の総和）
  twoPointVotes: number;               // 2点をつけた審査員数（笑いエフェクト判定用）
  judgeCount: number;                  // 採点時点の審査員母数
  laughTriggered: boolean;             // 過半数2点で笑いエフェクト発生したか
  moderation: "clean" | "flagged" | "hidden";
  reportCount: number;
  createdAt: string;
}

// 採点票（1審査員 → 1回答につき1票、0/1/2点）
interface Score {
  id: string;
  answerId: string;
  judgeParticipantId: string;          // 採点した審査員
  points: 0 | 1 | 2;
  createdAt: string;
  // UNIQUE(answerId, judgeParticipantId) で二重採点防止（上書きのみ）
}

// ユーザー（永続。ログイン登録者）
interface User {
  id: string;
  authProvider: "x" | "email";
  displayName: string;
  // --- 軸②：熟練度メーター → 段位（減らない経験値） ---
  rank: string;                        // 段位（例: "beginner".."master"）。§6.3
  masteryMeter: number;                // 熟練度メーター（経験値）
  // --- 軸①：ポイント（消費用通貨） ---
  points: number;                      // 消費可能なポイント残高
  // --- 実績 ---
  liveCount: number;                   // 参加回数
  awardCounts: { first: number; second: number; third: number };
  bestAnswerCount: number;
  createdAt: string;
}

// お題セット（ライブ1回ぶんの仕込み：組数 × 周回数 ＋ 予備）
interface TopicSet {
  id: string;
  name: string;
  topicIds: string[];
}

// ----- コレクション/ガチャ（抽象設計のみ。具体アイテム内容は将来課題） -----

// コレクションアイテム（種類・レアリティを抽象化）
interface CollectionItem {
  id: string;
  type: "character" | "costume" | "icon_part" | "bg_pattern"; // 拡張可
  rarity: "N" | "R" | "SR" | "SSR";    // 拡張可
  name?: string;                       // 具体名は将来定義
}

// ガチャの排出テーブル
interface GachaTable {
  id: string;
  name: string;
  cost: number;                        // 1回の消費ポイント
  entries: { itemId: string; weight: number }[]; // 排出重み
}

// ユーザーの所持・装備
interface UserInventory {
  userId: string;
  ownedItemIds: string[];
  equipped: {
    characterId?: string;
    costumeId?: string;
    iconPartIds?: string[];
    bgPatternId?: string;
  };
}
```

- **スコア算出**：ライブ内 = `Participant.scoreReceived`（自分の全 Answer.scoreTotal 合計）。個人順位 = ライブ内スコア降順で上位3名を表彰。ベストアンサー = 単一 Answer.scoreTotal 最多。
- **笑いエフェクト判定**：採点締切時 `twoPointVotes > floor(judgeCount / 2)` なら `laughTriggered = true`。
- **経済更新（`closed`時）**：
  - ポイント：`User.points += bonusByRank[順位]（+参加ボーナス）`（§5.4）。
  - 熟練度メーター：`User.masteryMeter += masteryGain(スコア, 順位, ベストアンサー)`（§6.2）→ 閾値超えで `rank` 昇段。
  - 実績カウント（`awardCounts` / `bestAnswerCount` / `liveCount`）更新。

## 16. 決定事項・未確定事項

### 16.1 本仕様での決定（改訂反映済み）
1. 進行＝各組90秒持ち時間、各人最大5回答、送信ごとに「タイマー停止→お題再提示→回答表示→10秒採点→再開」の審査サイクル、1件ずつ排他処理（§1）。
2. 周回制＝`roundsPerLive`（既定2、固定せず）、組ごと・周ごとに別お題（§1.3・§8）。
3. 結果＝全周終了後に**個人1〜3位を表彰**、順位別ボーナス（§1.7・§5.4）。
4. 審査＝1審査員1回答0/1/2点、過半数2点で笑いエフェクト、拍手/テンプレツッコミは純演出（§4）。
5. 経済＝**2軸**：消費用ポイント（＋ガチャ/コレクション抽象設計）と、熟練度メーター→段位（§5・§6）。
6. 段位＝**8段階の寄席モチーフ呼称で確定**（見習い→前座→二ツ目→真打→花形真打→大看板→名人→達人）（§6.3）。
7. お題＝事前登録＋司会半自動、**MVPは`text`のみ実装**（`format`フィールドは将来拡張を見込んで拡張可能に設計）（§8）。
8. 認証＝**ログイン必須（確定）**。ゲスト参加は一旦なし（`guestJoinAllowed` は既定値・現状運用ともにfalse固定）（§7）。
9. 演出＝幕間（暗転→タイトル）を進行に組込み、画面 L1b 追加。世界観は design-dept と別途（§12）。
10. 審査サイクルの尺＝**現行仕様のまま許容（確定）**。1組が最大25回の審査サイクルで90秒＋αに伸びる点は問題なしと判断。テンポ間引き等の緩和策は不要（§1.2）。
11. 見学者＝**採点不可（確定）**。見るだけの立場とし、笑いエフェクトの過半数判定の母数にも含めない（§7.3）。
12. 持ち時間＝**送信していない時間だけ進むpause方式で確定**。組の全員が最大回答数(5回)を使い切ったら、残り時間を待たず次へ進む早期終了ありで確定（§1.1）。
13. ポイント/メーターの係数＝**初期値確定**。表彰ボーナス(1位300/2位200/3位100/参加賞30/ベストアンサー+100pt)、熟練度メーター加算(参加+10/スコア1:1/順位1位+100・2位+60・3位+30/ベストアンサー+50)、8段位の昇段閾値(見習い0〜達人9,000)（§5.4・§5.5・§6.2・§6.3.5）。
14. ガチャ/コレクションの具体内容＝**衣装14種・アイコンパーツ10種・背景柄7種のラインナップ案、排出率案（推奨：N55%/R33%/SR10%/SSR2%）、段位との連動なし（全員同一排出率）で案を提示（§5.3.2）**。ラインナップの最終ビジュアル・排出率の最終値はdesign-dept／運用データを見て確定する前提。

### 16.2 ユーザーに追加確認したい論点
- **I. 排出率の最終値**：§5.3.2で提示した3案（A:カジュアル寄り／B:バランス型・推奨／C:渋め）のうちどれを採用するか。初期値はB案（N55%/R33%/SR10%/SSR2%）としてよいか。
- **J. GachaTableの構成単位**：type（costume/icon_part/bg_pattern）横断で1本のテーブルにするか、type別に複数テーブルを分けるか（§5.3.2④の初期案は「type横断で1本」）。
- **K. 段位連動**：§5.3.2⑤の推奨（連動なし＋段位到達時の無料ガチャチケット等で代替）でよいか、それとも段位が上がると排出率も優遇する方向を検討したいか。
